Introduction 
This document describes the HTTP-based communication between Archwave AES67 devices (from now on 
“device”) and their configuration peers (from now on “client”). These peer are most likely web clients of the page 
server by the own device.  
Note that the objects provided here might be used to configure connections with devices that are not made by 
Archwave. Particularly, the lists provided by the Archwave devices will include network devices and AES67 
streams which are present in the network, but which are not necessarily Archwave devices. The client cannot 
assume that any of the objects given here exists in an arbitrary AES67 device. E.g., the device list is provided 
so that the client can provide a link to the configuration website of all present AES67 devices.  
General 
1. Configuration parameters, option lists and commends are exchanged using:  
1.1. HTTP GET on pre-specified addresses, when data flows from device to client. 
1.2. HTTP POST on pre-specified addresses, when data flows from client to device. 
2. Data objects are transmitted using JSON (JavaScript Object Notation)

Commands 
These commands are provided by the device on fixed addresses. They are sent by the application to instruct 
the device to perform certain tasks. 
Reboot command 
Instructs the device to restart. A parameter can tell whether to restart normally or into the bootloader in order to 
perform a firmware update. 
- Location: <Device URL>/command 
- Notation: 
{ 
  "ArchwaveCMD": { 
    "version": 1, 
    "reboot": true, 
    "bootloader": false 
  } 
} 
 
Read-only objects 
These objects are provided by the device on fixed addresses. They represent the general status of the device 
or the network it is connected to. The client cannot change these; it reads them to inform the user or to provide 
some help with the setup. 
Device list 
Lists each AES67 device currently present in the network, with its name and  the address of its webserver. 
This object can change dynamically, the client should read it periodically to ensure a good user experience. 
Using an outdated list has technically no negative impact, no race conditions or similar problems are to be 
expected when updating this object or using its contents. 
- Location: <Device URL>/device_list 
- Notation: 
{ 
  "ArchwaveAES67": { 
    "version": 1, 
    "devices": [ 
      { 
        "hostname": "Device 1 hostname", 
        "IPaddress": "192.168.4.202:1234" 
      }, 
      { 
        "hostname": "Device 2 hostname", 
        "IPaddress": "192.168.4.203:8080" 
      } 
    ] 
  } 
}

Stream list 
Lists each AES67 stream currently provided in the network, with its name and its channel count.  
This object can change dynamically, the client should read it periodically to ensure a good user experience. 
Using an outdated list has technically no negative impact, no race conditions or similar problems are to be 
expected when updating this object or using its contents. 
- Location: <Device URL>/stream_list 
- Notation: 
{ 
  "ArchwaveAES67": { 
    "version": 1, 
    "streams": [ 
      { 
        "name": "Stream 1 name", 
        "channelCount": 8 
      }, 
      { 
        "name": "Stream 2 name", 
        "channelCount": 2 
      } 
    ] 
  } 
}

Parameter objects 
These objects are provided by the device on fixed addresses. They represent the configuration of different 
areas of the device. The client can change their contents, with the exception of the read -only fields (mainly 
status fields). Client changes on read-only fields are simply ignored.  
The status fields can change dynamically, the client should read them periodically.

Output channels 
This object contains the parameters that affect the audio that the device receives from the network. 
The amount of outgoing streams changes with the Audio Mode. This list should be read again after changing 
the Audio Mode parameter. 
The fields streamPath and manualSDP are mutually exclusive; if both of them are empty (“”) the channel is not 
connected; if one of them has content, it defines the way the connection is made; if both of them have content, 
the streamPath is used (Bonjour-based Ravenna connection). 
Channel number in both the streamPath and the manualSDPChannel fields are 1-based, i.e., the first channel is 
the number 1. 
- Location: <Device URL>/outputs 
- Notation: 
{ 
  "ArchwaveAES67": { 
    "version": 1, 
    "outputs": [ 
      { 
        "name": "Analog Output 1 on <device name>", 
        "streamPath": “Digital Inputs on MyDevice/4”, 
        "manualSDP": “”, 
        "manualSDPChannel": 0, 
 
      }, 
      { 
        "name": "Analog Output 2 on <device name>", 
        "streamPath": “Analog Inputs on Device 8/4”, 
        "manualSDP": “”, 
        "manualSDPChannel": 0, 
 
      }, 
      { 
        "name": "Digital Output 1 on <device name>", 
        "streamPath": “Digital Inputs on MyDevice/5”, 
        "manualSDP": “”,

      }
